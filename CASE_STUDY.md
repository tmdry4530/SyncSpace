# SyncSpace — 케이스 스터디

> 채팅과 공동 문서를 한 화면에 둔 실시간 협업 워크벤치. 솔로로 설계·구현·배포·운영까지.

- **라이브 데모**: https://server-production-cc9e.up.railway.app (가입 후 바로 사용 가능)
- **역할**: 1인 (설계, 프론트엔드, 백엔드, 인프라, 디자인 전부)
- **한 줄**: Supabase + Vercel + Railway로 시작한 3-서비스 구조를 **단일 Railway 서비스**(SPA + REST + Yjs WebSocket + Postgres)로 통합하고, 사람 세션 인증과 다크모드 디자인 시스템까지 직접 구현한 프로젝트.

이 문서는 기능 목록이 아니라 **어떤 결정을 왜 내렸고, 무엇이 어려웠고, 다시 하면 무엇을 바꿀지**를 정리한 기록입니다.

---

## 1. 무엇을 만들었나

팀의 협업은 보통 "채팅에서 논의 → 결정 → 다시 문서로 옮김"으로 쪼개지고, 그 과정에서 맥락이 끊깁니다. SyncSpace는 **채팅 패널과 공동 문서 편집기를 한 화면(워크벤치)에 두어**, 대화 중 나온 내용을 바로 문서로 정리하도록 만든 앱입니다.

- 채널 채팅 + Yjs/TipTap 기반 공동 문서 편집을 좌우로 배치
- 워크스페이스 / 채널 / 문서를 URL에 담아 새로고침 후에도 맥락 복원
- 실시간 동시 편집, 접속 인원(presence), 라이트/다크 테마

---

## 2. 아키텍처: 3-서비스 → 단일 서비스

**처음 (BaaS 조합):**

```
Vercel(프론트) ── Supabase(Auth + Postgres + RLS + Realtime)
       └──────── Railway(Yjs WebSocket 서버)
```

**지금 (Railway 단일 서비스):**

```
Railway 1개 서비스
  └ Node HTTP 서버: 빌드된 SPA 정적 서빙 + REST API + Yjs WS 업그레이드
Railway Postgres: 데이터 + 세션 + Yjs 스냅샷
```

세 곳에 흩어진 운영(배포 2곳, 인증/DB 1곳)을 한 서비스 + 한 DB로 합쳤습니다. 한 번의 `pnpm build`가 SPA를 `dist/`로, 서버를 `server/dist/`로 빌드하고, 같은 Node 서버가 정적 파일과 API와 WebSocket을 한 포트에서 처리합니다.

---

## 3. 핵심 결정과 이유

**① 실시간의 중심을 Yjs(CRDT)로.**
채팅과 문서 둘 다 별도의 Yjs room으로 처리했습니다. 문서 동시편집은 CRDT가 충돌 해결을 보장하고, 채팅도 같은 채널을 재사용해 인프라를 단순화했습니다. 서버는 각 room의 변경을 Postgres 스냅샷으로 영속화합니다.

**② 상태 책임을 3계층으로 분리.**
하나의 전역 스토어에 다 넣지 않고 성격별로 나눴습니다: `Zustand`(UI 상태) / `TanStack Query`(서버 목록·메시지 캐시, 폴링) / `Yjs`(실시간 room·presence). 덕분에 컴포넌트가 UI 조작·서버 캐시·WS 동기화를 한꺼번에 떠안지 않습니다.

**③ 라이브는 WebSocket, 서버 상태는 폴링.**
메시지 히스토리·채널·문서 목록은 1.5초 TanStack 폴링으로 신선도를 유지하고, 실제 라이브(채팅 입력·문서 편집·presence)는 Yjs WS로 흐르게 했습니다. 단순하지만 명확한 경계.

**④ Supabase Auth → 자체 세션 인증.**
Supabase를 걷어내면서 인증을 직접 구현했습니다. JWT가 아니라 **불투명(opaque) 랜덤 토큰**을 쓰고, 토큰 자체가 아니라 `SHA-256(token + 서버 시크릿)` 해시만 DB에 저장합니다. 비밀번호는 scrypt(per-hash salt), 세션은 `HttpOnly` 쿠키(SameSite=Lax). 토큰을 폐기/회전할 수 있게(서버가 무효화 가능) opaque 토큰을 택했습니다.

**⑤ 다크모드 = 시맨틕 토큰 아키텍처.**
색을 컴포넌트에 직접 박지 않고 `:root`에 시맨틱 토큰(약 79개)으로 정의하고, `:root[data-theme='dark']` 한 블록만 토큰 값을 재정의하면 앱 전체가 한 번에 뒤집히게 설계했습니다. 라이트/다크/시스템 + 새로고침 시 깜빡임 없는 사전-페인트 초기화.

## 4. 어려웠던 것 (디버깅 전쟁담)

**CORS 누수로 프로덕션이 백지.**
Railway로 컷오버한 직후, 라이브가 빈 화면이었습니다. 헤드리스 브라우저로 출처별로 같은 에셋을 때려보니 — 자기 도메인 출처는 **403**, `localhost`는 200. 원인: 환경변수를 `--skip-deploys`로 설정해서 **이미 떠 있는 컨테이너에 반영되지 않았고**, 그 컨테이너의 CORS 허용목록이 기본값(localhost)뿐이었습니다. 교훈: 환경변수는 부팅 시점에 읽힌다. 설정 후 반드시 재배포. (재배포 한 번으로 해결.)

**빌드타임 env와 도메인의 닭-달걀.**
`VITE_*`(API/WS URL)는 빌드 시점에 번들에 구워집니다. 그런데 도메인은 서비스를 만든 뒤에야 발급됩니다. 순서를 틀리면 SPA가 localhost를 부릅니다. "서비스 생성 → 도메인 발급 → 도메인 의존 변수 설정 → 그 다음 빌드"로 순서를 고정해야 했습니다.

**빌드는 통과하는데 화면만 깨지는 다크모드 누수.**
다크에서 채팅창만 흰색으로 남았습니다. 레거시 클래스(`.message-list { background:#f7f9fb }`)와 토큰 기반 클래스(`.ap-wb-chat-scroll`, 배경 미지정)를 **이중 적용**했는데, 토큰 클래스가 배경을 안 덮으니 하드코딩된 라이트 색이 그대로 보인 것. `tsc`·`vite build`·테스트는 전부 통과했습니다. **이건 빌드가 아니라 "라이브에서 다크모드로 직접 보는 감사"로만 잡혔습니다.** 시각 버그는 시각으로 검증해야 한다는 걸 재확인.

---

## 5. 다시 하면 바꿀 것 (회고)

- **수요 검증을 먼저.** 가장 큰 교훈은 코드가 아닙니다. 만들기 전에 "이걸 절실히 원하는 사람"을 먼저 찾았어야 합니다. (붐비는 시장 + 검증 없는 빌드 = 전형적인 "문제 없는 해결책".)
- **CSS를 토큰 일원화.** 레거시 클래스와 토큰 클래스를 이중 적용한 게 다크모드 누수의 근본 원인. 처음부터 토큰만 소비하는 단일 체계였다면 누수가 구조적으로 불가능했을 것.
- **빌드타임 env 결합 줄이기.** `VITE_*` 의존을 줄이고 같은-출처 상대경로 + 런타임 설정으로 갔다면 배포가 훨씬 덜 부서졌을 것.
- **채팅 실시간 일원화.** 지금은 히스토리 폴링 + Yjs WS 병행. SSE나 WS 하나로 합치면 중복 부하/복잡도가 줄어듦.

---

## 6. 검증 방식

"빌드 통과"를 완료로 보지 않았습니다. 배포할 때마다 **실제 라이브 도메인을 헤드리스 브라우저로 직접 검증**했습니다: 가입/로그인 왕복, 워크스페이스·채널·문서 생성, 채팅 전송, 문서 편집, WebSocket 연결, 다크모드 전환, 세션 유지(새로고침), 네트워크 탭에서 실패 요청·잘못된 호스트 호출 확인. CORS 백지와 다크모드 누수 둘 다 이 방식으로 잡혔습니다.

## 7. 기술 스택

| 구분 | 기술 |
| --- | --- |
| Frontend | React 19, TypeScript, Vite |
| Routing / State | React Router 7, Zustand(UI), TanStack Query(서버) |
| Realtime / Editor | Yjs, y-websocket, TipTap(Collaboration) |

| DB / Deploy | Railway Postgres, Railway 단일 서비스 |
| Design | 시맨틱 CSS 토큰 + 라이트/다크/시스템 테마 |
