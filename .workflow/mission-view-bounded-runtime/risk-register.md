# Phase 6 — Bounded coding runtime: risk register (MVP 2.2)

Companion to `runtime-design.md`. Severity = pre-mitigation impact if exploited.
Residual = remaining exposure AFTER the mitigation, in the smallest (flag-OFF, no-container) version.
"Gate" = the verification step that proves the mitigation holds before the runtime can ship enabled.

Global control: **`MISSION_RUNTIME_ENABLED` defaults OFF**; selection requires `AGENT_RUNTIME_MODE=live`
AND the flag AND a coding-marked task (`runtime-design.md` §1). With the flag off, every risk below is
**not reachable** — the worker path is identical to today.

| ID | Risk | Severity | Mitigation (file-level) | Residual risk | Verifying gate |
|----|------|----------|-------------------------|---------------|----------------|
| R-1 | **Path traversal / jail escape** — read/write outside the task workspace via `..`, absolute path, or in-jail symlink → off-jail target | Critical | `resolveInsideJail` (`workspace.ts`, design §2): reject absolute/NUL; `fs.realpath` collapses symlinks+`..`; `path.relative` segment containment (not `startsWith`); write-create realpaths deepest existing ancestor; optional `O_NOFOLLOW` on write open. Worktree under `os.tmpdir()`, never under source tree. | Low. realpath closes symlink+`..`; one residual is a TOCTOU symlink-swap between resolve and open — covered by `O_NOFOLLOW` if enabled, otherwise a sub-100ms race on a single-tenant worker. | Gate 7: tests assert `../`, absolute, NUL, and symlink-to-outside are all rejected for read AND write. |
| R-2 | **Command injection** — model-influenced string reaches a shell | Critical | argv-array `execFile`/`spawn`, `shell:false`; **fixed command table keyed by an enum** — model supplies only a key, never argv (`commandRunner.ts`, design §3). No string concatenation reaches a process. | Negligible. No shell is ever invoked; no metacharacter path exists. | Gate 7: test that a request with a non-table key is rejected pre-spawn; static check that no code path passes a model string to `exec`/`sh -c`. Gate 6: dual-Opus reviews the allowlist + spawn discipline. |
| R-3 | **Secret / env exfiltration** — child reads `ANTHROPIC_API_KEY`, `DATABASE_URL`, `AUTH_SECRET`, `AGENT_TOKEN_PEPPER`, `SUPABASE_*` from inherited env and exfiltrates | Critical | Child env **constructed from empty** with a 7-key allow list (`PATH`,`HOME`,`LANG`,`LC_ALL`,`TMPDIR`,`TZ`,`TERM`); `HOME=jail`; secrets never on argv (`commandRunner.ts` `buildChildEnv`, design §4). | Low. Secrets are absent from the child env entirely. Residual only if a future edit re-introduces `...process.env` — caught by the regression test. | Gate 7: unit test asserts constructed child env contains NONE of the named secret keys. Gate 6: dual-Opus confirms no-creds boundary. |
| R-4 | **Writing outside the jail / corrupting the main checkout** — direct write or git op mutates `missionRepoRoot` | High | All writes via `resolveInsideJail`; only read-only `git status/diff` allowlisted (no `add/commit/checkout/reset/clean`); worktree `--detach` (no branch moves); `core.hooksPath=/dev/null` (design §2,§3,§6). NEVER writes back to main checkout. | Low. The main checkout's working tree is a different directory; git writes are not allowlisted. | Gate 7: after a run, assert `git -C missionRepoRoot status` is unchanged and no file under the source tree was modified. Gate 6: dual-Opus "no-main-mutation" check. |
| R-5 | **Network egress by child / dependency script** — exfiltration, SSRF, fetching a payload | High | `npm_config_offline=true`, `pnpm install` NOT allowlisted, allowlisted commands confirmed to run offline (design §4). | **Accept-with-monitoring (irreducible).** A dependency's own postinstall/test script can still attempt egress; Node has no per-process net namespace without a container (contract: no Docker). This is the headline residual and the reason for flag-OFF-by-default. | Gate 7: run each allowlisted command with network disabled + `npm_config_offline=true` and confirm green (proves no *required* egress). Gate 6: reviewers explicitly accept or reject this residual — if rejected, runtime stays design-only. |
| R-6 | **Resource exhaustion** — infinite loop, fork bomb, huge writes, log flood (CPU/mem/disk/time) | High | Per-command `timeout`+`SIGKILL` (120s), `maxBuffer`=64 KiB, loop `maxIterations`=12 + `wallClockMs`=300s + worker `AbortSignal`; bounded tail capture; temp tree removed at teardown (design §3,§5.3,§7). | Medium. Bounds a single command + the loop, but not a forked grandchild's aggregate CPU/mem (cgroups are container-only). A 120s CPU burn per command is possible but time-boxed and single-tenant. | Gate 7: test that a sleeping/looping command is SIGKILLed at the timeout and that oversized output is truncated to the tail. |
| R-7 | **Prompt-injected destructive command** — task text or file content instructs "rm -rf /", "git push", "curl evil" | High | Model **cannot express** arbitrary commands — only an allowlist key; the table has no destructive/network/VCS-write entry (design §3,§7,threat T-7). | Negligible for destructive ops; worst case is repeated `git:diff` (bounded by R-6). | Gate 7: adversarial test feeds injection text; assert no command outside the table runs. Gate 6: dual-Opus reviews allowlist completeness. |
| R-8 | **Worktree teardown failure leaves orphan worktree / temp tree** (disk leak, stale registration) | Medium | Teardown in `finally` (always runs): `git worktree remove --force` + `worktree prune` + `fs.rm(recursive)`; optional startup sweep of stale `mission/*` worktrees (design §2,§7). | Low. A hard crash mid-run could orphan one temp dir until the next `prune`/sweep; disk-only, no correctness/security impact. | Gate 7: test teardown removes the worktree on success AND when the run throws; simulate crash → next-run prune clears it. |
| R-9 | **Shared `.git/worktrees/<id>` admin-dir coupling** — perturbing parent repo state via the linked-worktree admin dir | Medium | Jail root is the worktree working tree; the linked `.git` is a *file* pointer and `resolveInsideJail` blocks climbing to the real `.git`; read-only git via `-C` with hooks disabled (design §2 W-1, threat T-8). | Low. The shared object store is append-only via our read-only git; no allowlisted op writes refs/objects. | Gate 7: assert no refs/objects in `missionRepoRoot/.git` change after a run. |
| R-10 | **Provider tool-use loop / cost blowup** — model emits tool calls indefinitely | Medium | `maxIterations`+`wallClockMs`+optional token cap; per-turn `timeoutMs`; `ctx.signal` re-checked each iteration (design §5.3). | Low. Cost is bounded by iteration×maxTokens; a misbehaving model is cut off deterministically. | Gate 7: test the loop terminates at `maxIterations` and at `wallClockMs`. |
| R-11 | **Half-configured runtime runs** — flag on but mode≠live, or git/pnpm/repoRoot missing | Medium | `assertMissionRuntimeConfig` in `readConfig` throws at boot on any inconsistency; binary paths resolved once at startup (design §1,§9). | Negligible. Fail-closed at boot, mirroring `assertAgentRuntimeConfig` (`config.ts:124-132`). | Gate 7: boot-assertion tests for each missing-precondition case. |
| R-12 | **Regression: coding runtime leaks into normal chat/doc agents** — existing roles enter the spawn/jail path | High | Selection is a strict conjunction (flag + live + coding-marked task); recommend a dedicated `mission-coder` role so existing roles are provably untouched (design §1.2). | Low. Default-off + explicit task marker; existing role behavior is unchanged when flag off. | Gate 7: test that with the flag OFF (and with it ON but task unmarked) `getAgentRuntime` returns the same runtime as today for planner/reviewer/etc. Full `verify:all` + server suite. |
| R-13 | **False engineering events** — emitting `vcs_event` commit/PR or claiming execution that didn't happen | Low | Runtime emits only events backed by real actions; no `commit`/`pr_opened` `vcs_event` in 2.2 (no real commit/PR); demo events carry `demo:true` (design §6; contract "no fake claims"). | Negligible. | Gate 5 (demo honesty) + Gate 7 review: assert runtime emits no commit/PR `vcs_event`. |

## Residual-risk summary

- **Single biggest residual: R-5 (network egress by a dependency's own build/test script).** It is
  irreducible in the smallest version because true egress prevention needs a container/namespace, which
  the contract excludes. It is mitigated to "no *required* network" (offline pnpm, no install) and is the
  primary justification for `MISSION_RUNTIME_ENABLED` defaulting OFF and for gating the whole runtime
  behind a dual-Opus review (Gate 6) and a proven rollback (Gate 7).
- All Critical risks (R-1/R-2/R-3) reduce to **Low/Negligible** residual via realpath jailing, argv-array
  + enum-keyed allowlist, and an empty-baseline child env — each with a dedicated Gate-7 regression test.
- If Gate 6 reviewers decline to accept R-5's residual, the correct outcome is **runtime stays
  design-only**; Mission View + the 7 events + the demo already deliver the product without it.
